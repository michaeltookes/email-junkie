### ~~7: Draft generation engine~~
**Resolved**: 2026-07-21 (branch thread-quote)
**Description**: Completed the draft engine by giving the model real **thread/quote context** instead of only the single incoming message. A new pure `EmailThreadParser` splits a plain-text body at the first quoted-reply marker (`>` prefix, Outlook `-----Original Message-----`, or a Gmail/Apple `On <date> … wrote:` attribution) into the sender's **latest** text and the **quoted history** beneath it, and exposes a readable, de-quoted, bounded form of that history for prompting. `VoiceProfiler`'s inline quote-stripping now delegates to the same parser, so quote detection has a single source of truth. `DraftGenerator` builds a **structured** reply prompt — the fresh message to reply to, plus an "earlier in this thread (context only — do not quote it back)" section when history exists — with the fresh text and the history budgeted **separately** (`maxIncomingChars` / `maxThreadChars`), so a long quoted tail can no longer truncate away the actual new message; the system prompt now tells the model to use earlier messages as context but reply only to the latest. Both the on-demand Settings draft and the watcher-enqueued draft benefit, since they share `DraftGenerator`. Earlier-delivered pieces of this item remain in place: voice-profile injection via `VoiceProfile.promptBlock()` (neutral fallback), `Subject:`-line output cleaning, provider/rate-limit errors surfaced through `AppState.draftMessage(for:)`, the draft tied to its source message (UID + UIDVALIDITY, subject, sender, `Re:` subject) for correct threading, and the live-verification (2026-07-19) that a reply in the user's voice was generated and `In-Reply-To`/`References` threading works. 18 new tests (9 `EmailThreadParser` split/readable-history cases, 5 `DraftGenerator` thread-context cases incl. fresh-message-not-starved and per-section caps, plus preserved voice/cleaning coverage). Thread context is drawn from the message's own quoted history (no extra IMAP round-trip); fetching sibling thread messages over IMAP by `References` is a possible future enhancement. Inline draft editing is item 19; low-confidence "needs info" handling is item 13.

### ~~41: Yahoo / AT&T (att.net) & provider-agnostic mailbox support~~
**Resolved**: 2026-07-21 (branch provider-folders)
**Description**: Removed the Gmail-only special-folder assumption so the IMAP path works on Yahoo-backed providers (AT&T `att.net`, Yahoo, AOL, etc.). Added `MailboxNaming` (a Sendable value holding the account's `sent`/`drafts`/`allMail`/`junk` folder names, with `.gmail`, `.yahoo`, and `.generic` layouts, a `supportsAllMail` flag, and `forHost(_:)` resolution that maps Gmail/Yahoo-backed hosts — `att.net`, `sbcglobal`, `bellsouth`, `aol`, `ymail`, `rocketmail` — to the right layout and falls back to generic). `MailAccountCredentials` exposes `mailboxNaming` derived from its host, and `Mailbox.imapName(using:)` resolves the live server folder (with `.allMail` falling back to `INBOX` when the provider has none). The four IMAP handlers (fetch, body-fetch, search, append) now target per-account folder names, which fixes voice sampling from Sent, save-as-draft `APPEND`, and All-Mail search on non-Gmail accounts; the param-less `Mailbox.imapName` remains a stable identity/label for persisted source-mailbox tags and reply-dispatch guards. On the app side, `AppState.suggestedIMAPHost(forEmail:)` maps known domains to their IMAP host and `applySuggestedHostIfDefault()` auto-fills the host from the email domain (never overwriting a hand-entered custom host), wired into the email field in onboarding and Settings; the browser's "All Mail" folder option is now shown only when the connected provider actually has one (`supportsAllMailFolder`). Connection primitives were already generic (user-settable host + `imap.`→`smtp.` SMTP derivation), so this filled the remaining provider-specific gaps. 16 new tests (8 package: host→layout mapping, per-provider folder resolution, All-Mail fallback, credentials integration; 8 app: host suggestion, default-vs-custom auto-fill, All-Mail gating). Remaining as follow-ups: server-side special-folder discovery via IMAP `LIST` SPECIAL-USE (RFC 6154) for arbitrary providers, an explicit per-folder user override in Settings, documenting the Yahoo app-password / AT&T secure-mail-key setup, and live-verification against a real `att.net` account. The high-volume cleanup mechanism this unblocks across providers is item 42.

### ~~39: Server-side inbox search & filtered/paged fetch~~
**Resolved**: 2026-07-19 (branch inbox-search)
**Description**: Added a `searchMessages` capability to `MailProvider` so any message can be found across a full mailbox rather than only showing the last N. A `MailSearchCriteria` value (free-text/`from`/`subject` keywords, `since`/`before` dates, read state, flagged) resolves to an IMAP `SearchKey` (fields AND-ed; `.all` when empty), and `IMAPMailProvider` drives `LOGIN → SELECT → UID SEARCH → paged UID FETCH (envelopes) → LOGOUT` on SwiftNIO. Because `SEARCH` runs server-side, only matching UIDs return; a bounded page of those UIDs is then fetched, so large/messy mailboxes never download in full. `MailSearchResult` carries the page (newest first), the total match count, the offset, and a `hasMore` flag; empty results and offsets past the end short-circuit without a fetch. Paging (`MailSearchPaging`) is a pure, unit-tested function and criteria→`SearchKey` encoding is factored into static helpers. Added `Mailbox.allMail` (`[Gmail]/All Mail`) as the broadest search target. Search is a default-unsupported protocol method (like `sendMessage`) so existing conformers are unaffected. 17 tests: paging edge cases, criteria encoding + `ALL` fallback, the full `SEARCH`/`FETCH` state machine, empty/second-page/offset-past-end results, mailbox targeting, and login/search failures. The consuming pop-out browser UI (search field + filters + folder picker) is item 40; `has-attachment` (`BODYSTRUCTURE`) and Gmail `X-GM-RAW` remain optional stretch enhancements.

Scheduled re-learn is item 20; empty/sparse Sent cold-start is item 25.

### ~~8: Native macOS notification approval UX~~
**Resolved**: 2026-07-14 (branch approval-ux)
**Description**: Surfaced watcher-produced drafts for one-tap approval. When a draft is enqueued, a native `UNUserNotificationCenter` notification fires with inline **Approve** and **Deny** actions (both `authenticationRequired`; Deny is also `destructive`); tapping the body opens the review window. A `DraftNotifying` protocol wraps the notification center behind an injectable seam — `UserNotificationService` (registers the `DRAFT_READY` category, requests authorization at launch, posts/removes by the draft's stable `identity`, and routes action responses back on the main actor) with a `NullDraftNotifier` default so constructing an `AppState` never touches the notification system, and a `FakeDraftNotifier` for tests. A SwiftUI **Review Drafts** window lists every pending draft with the incoming message and the proposed reply side by side, per-draft Approve/Deny with an in-flight spinner, and a header showing what Approve will do (send vs save) — which also closes item 9's deferred approval-indicator criterion. `approveDraft`/`denyDraft` operate on individual queue members (multiple pending drafts are individually actionable): Approve blocks drafts from a different connected account, keeps the draft durably queued while `performSend`/`performSave` is in flight, records a durable approved-draft tombstone after success before cleanup is treated as complete, and filters tombstoned drafts on launch so stale `PendingDrafts.json` contents cannot be approved again; Deny durably discards without sending. `Draft` gained a persisted, truncated `incomingBody` (captured during drafting) and a scoped `identity`. Opened from a menu-bar "Review Drafts (N)…" item (⌘R) and from the notification's open action. 37 new tests (approval/deny, account mismatch, in-flight durability, approved tombstone cleanup, notification-action routing, enqueue-posts-notification with incoming-body capture); notification test double split into its own file to respect length limits.

### ~~5: Inbox watcher~~
**Resolved**: 2026-07-13 (branch inbox-watcher)
**Description**: Added a timer-driven inbox watcher that polls while the Mac is awake and turns new, replyable messages into queued drafts. `InboxWatcher` (the scheduling half) owns a repeating `Timer` and `NSWorkspace` sleep/wake handling — it pauses the timer on sleep and resumes with an immediate catch-up poll on wake, single-flighting ticks so polls can't stack. The poll *policy* lives on `AppState` (`pollInboxOnce`): fetch the recent inbox (envelope-level), and for each message that is new and replyable — oldest first, so drafts read chronologically — generate a reply via the existing draft engine and enqueue it to `pendingDrafts` (driving `pendingDraftCount`). A message is marked processed the moment it is selected for drafting, so it is never drafted twice even across restarts (transient-failure retry is deferred to item 27). Processed identity uses the RFC 5322 Message-ID when present, falling back to `UIDVALIDITY:UID`; the bounded (1000-key, oldest-evicted) `ProcessedMessages` set persists to `ProcessedMessages.json`. Watching auto-starts at launch when an account + LLM are connected, is pausable from a new menu-bar "Start/Pause Watching" control, reschedules on a poll-interval change, and stops on disconnect. Reply-worthiness is a light gate for now (real sender, not self) — fuller filtering is item 17. 19 new tests (9 `ProcessedMessages`, 10 watcher: dedup across polls, chronological enqueue, skip-self/processed, fetch-error, mark-processed-on-failure, lifecycle transitions). Refactored the message-preview actions into `AppState+MessagePreview.swift` to keep `AppState.swift` within the file-length limit.
