### ~~8: Native macOS notification approval UX~~
**Resolved**: 2026-07-14 (branch approval-ux)
**Description**: Surfaced watcher-produced drafts for one-tap approval. When a draft is enqueued, a native `UNUserNotificationCenter` notification fires with inline **Approve** and **Deny** actions (Approve is `authenticationRequired`, Deny `destructive`); tapping the body opens the review window. A `DraftNotifying` protocol wraps the notification center behind an injectable seam — `UserNotificationService` (registers the `DRAFT_READY` category, requests authorization at launch, posts/removes by the draft's stable `identity`, and routes action responses back on the main actor) with a `NullDraftNotifier` default so constructing `AppState` never touches the notification system, and a `FakeDraftNotifier` for tests. A SwiftUI **Review Drafts** window lists every pending draft with the incoming message and the proposed reply side by side, per-draft Approve/Deny with an in-flight spinner, and a header showing what Approve will do (send vs save) — which also closes item 9's deferred approval-indicator criterion. `approveDraft`/`denyDraft` operate on individual queue members (multiple pending drafts are individually actionable): Approve dispatches through the shared `performSend`/`performSave` per the send-behavior setting and removes the draft from the persisted queue on success; Deny discards. The queue removal is best-effort-persisted so an approved draft can never be re-approved. `Draft` gained a persisted, truncated `incomingBody` (captured during drafting) and a scoped `identity`. Opened from a menu-bar "Review Drafts (N)…" item (⌘R) and from the notification's open action. 33 new tests (approval/deny, notification-action routing, enqueue-posts-notification with incoming-body capture); notification test double split into its own file to respect length limits.

### ~~5: Inbox watcher~~
**Resolved**: 2026-07-13 (branch inbox-watcher)
**Description**: Added a timer-driven inbox watcher that polls while the Mac is awake and turns new, replyable messages into queued drafts. `InboxWatcher` (the scheduling half) owns a repeating `Timer` and `NSWorkspace` sleep/wake handling — it pauses the timer on sleep and resumes with an immediate catch-up poll on wake, single-flighting ticks so polls can't stack. The poll *policy* lives on `AppState` (`pollInboxOnce`): fetch the recent inbox (envelope-level), and for each message that is new and replyable — oldest first, so drafts read chronologically — generate a reply via the existing draft engine and enqueue it to `pendingDrafts` (driving `pendingDraftCount`). A message is marked processed the moment it is selected for drafting, so it is never drafted twice even across restarts (transient-failure retry is deferred to item 27). Processed identity uses the RFC 5322 Message-ID when present, falling back to `UIDVALIDITY:UID`; the bounded (1000-key, oldest-evicted) `ProcessedMessages` set persists to `ProcessedMessages.json`. Watching auto-starts at launch when an account + LLM are connected, is pausable from a new menu-bar "Start/Pause Watching" control, reschedules on a poll-interval change, and stops on disconnect. Reply-worthiness is a light gate for now (real sender, not self) — fuller filtering is item 17. 19 new tests (9 `ProcessedMessages`, 10 watcher: dedup across polls, chronological enqueue, skip-self/processed, fetch-error, mark-processed-on-failure, lifecycle transitions). Refactored the message-preview actions into `AppState+MessagePreview.swift` to keep `AppState.swift` within the file-length limit.

### ~~10: Secure local storage~~
**Resolved**: 2026-07-02 (branch secure-local-storage)
**Description**: Added a protocol-based `SecretStore` with a `KeychainStore` (macOS Keychain generic-password items, after-first-unlock/device-only, no entitlement) and an `InMemorySecretStore` for tests/previews, plus typed `SecretKey`s for Gmail tokens, BYO Google client credentials, and per-provider LLM keys. Secrets never touch the plaintext settings file; app state stays in local JSON. Added a Privacy section to Settings stating the local-first posture. 23 tests (in-memory and fake-Keychain behavior tests always run; real-Keychain tests probe and skip where unavailable). Fixed `removeAll` to loop `SecItemDelete` (macOS deletes one match per call), and Keychain replacements now update existing items in place so a failed save preserves the previous secret. Note: the disconnect UI lands with item 3 and the onboarding privacy display with item 2; the storage primitives and Settings statement are done.
